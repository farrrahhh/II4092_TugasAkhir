\section{Gambaran Umum Sistem}

Bagian ini memaparkan gambaran umum sistem Teman Belajar ASD melalui dua model perancangan: \textit{use case diagram} yang mendeskripsikan interaksi antara pengguna dan sistem, serta \textit{sequence diagram} yang menggambarkan urutan komunikasi antar komponen untuk setiap fungsionalitas utama. Kedua model ini menjadi dasar implementasi sistem secara keseluruhan.

\subsection{Use Case}

Pada sistem aplikasi pembelajaran Algoritma dan Struktur Data berbasis \textit{guided learning} ini, terdapat satu aktor yaitu \textit{user} yang merupakan mahasiswa sebagai pengguna sistem. \textit{User} berinteraksi dengan sistem untuk mempelajari materi ASD secara terstruktur, mulai dari memilih topik, memahami konsep, melihat contoh, mengerjakan latihan, hingga memantau progres belajar.

Sistem dirancang sebagai \textit{teman belajar digital} yang mendampingi mahasiswa dalam proses pembelajaran. Melalui integrasi LLM, sistem mampu memberikan penjelasan materi, contoh implementasi, serta umpan balik terhadap latihan yang dikerjakan oleh \textit{user} secara interaktif dan adaptif.

Tabel \ref{tab:use-case} menunjukkan bahwa terdapat satu aktor yaitu \textit{user} yang dapat menjalankan berbagai aktivitas untuk memenuhi kebutuhan fungsional sistem. Proses pembelajaran dimulai dari \textit{user} melakukan autentikasi (UC-01), kemudian memilih topik ASD yang ingin dipelajari (UC-02).

Selanjutnya, \textit{user} akan memasuki alur pembelajaran terstruktur dengan melihat materi (UC-03) yang berisi penjelasan konsep yang dihasilkan sistem. Untuk memperdalam pemahaman, \textit{user} dapat melihat contoh implementasi (UC-04) sebelum mengerjakan latihan yang diberikan (UC-05).

Setelah menyelesaikan latihan, \textit{user} dapat melihat pembahasan (UC-06) sebagai bentuk umpan balik dari sistem. Relasi \textit{extend} antara UC-05 dan UC-06 menunjukkan bahwa pembahasan hanya muncul setelah \textit{user} mengerjakan latihan. Selain itu, \textit{user} juga dapat melihat ringkasan materi (UC-07) untuk memperkuat pemahaman, serta memantau progres belajar melalui fitur \textit{progress} (UC-08).

Dengan demikian, seluruh \textit{use case} yang dirancang telah mencerminkan alur \textit{guided learning} yang mendukung pembelajaran ASD secara terstruktur dan interaktif, serta memenuhi kebutuhan fungsional sistem.

\subsection{Arsitektur Sistem}

Sistem Teman Belajar ASD dirancang menggunakan arsitektur berlapis yang terdiri atas empat komponen utama: \textit{Client}, \textit{Frontend}, \textit{Backend}, serta \textit{Database} dan OpenAI API. \textit{Client} mengakses sistem melalui \textit{browser} atau perangkat \textit{mobile} via HTTPS. \textit{Frontend} yang dibangun dengan Next.js berperan sebagai antarmuka pengguna sekaligus \textit{proxy} ke \textit{backend}, menangani halaman autentikasi, \textit{dashboard} topik, alur pembelajaran, dan \textit{progress tracker}. \textit{Backend} berbasis FastAPI mengelola seluruh logika bisnis meliputi autentikasi, progres belajar, generasi soal latihan, evaluasi jawaban, dan \textit{adaptive engine}. Data pengguna disimpan di PostgreSQL, sedangkan generasi konten pembelajaran dilakukan melalui OpenAI API (GPT-4o-mini). Seluruh layanan di-\textit{deploy} di GCP Cloud Run dengan \textit{pipeline} CI/CD melalui GitHub Actions.

\section{Perancangan \textit{Website}}

\subsection{Perancangan Arsitektur Sistem}

Aplikasi Teman Belajar ASD dibangun menggunakan arsitektur \textit{client-server} yang terdiri dari tiga lapisan utama: \textit{frontend}, \textit{backend}, dan \textit{database}. Komponen \textit{frontend} dibangun dengan Next.js dan berkomunikasi dengan \textit{backend} melalui RESTful API berbasis FastAPI. \textit{Backend} menangani logika bisnis, autentikasi JWT, serta integrasi dengan OpenAI API untuk menghasilkan konten pembelajaran secara adaptif. \textit{Deployment} dilakukan secara otomatis melalui integrasi dengan repositori Git. Setiap \textit{push} ke \textit{branch} utama akan memicu \textit{pipeline} GitHub Actions yang mendeteksi perubahan pada direktori \texttt{frontend/} atau \texttt{backend/} secara terpisah. \textit{Image} Docker di-\textit{build} dan dikirim ke Google Artifact Registry, kemudian di-\textit{deploy} ke layanan Cloud Run yang sesuai. Autentikasi ke Google Cloud menggunakan Workload Identity Federation sehingga tidak memerlukan \textit{service account key} yang disimpan secara statis.
